# SOBI CRM — یک سیستم‌عامل ماژولار کسب‌وکار

🌐 **زبان:** [English](README.md) · [Deutsch](README.de.md) · **فارسی**

SOBI CRM یک **سیستم‌عامل کسب‌وکار** چندمستأجری، متادیتامحور و مبتنی بر هوش
مصنوعی است. CRM قابلیت اصلی آن است که روی یک پلتفرم مشترک ساخته شده، جایی
که هر مستأجر فقط ماژول‌های صنعتی مرتبط با کسب‌وکار خودش را فعال می‌کند.

این یک نسخه‌ی نمایشی نیست — یک بنیان واقعی و در سطح تجاری است: وضعیت
امنیتی و انطباق با GDPR قوی، قابلیت پیکربندی کم‌کد (سازنده‌ی فرم / گردش‌کار
/ موجودیت / داشبورد)، یک موتور اتوماسیون، یک لایه‌ی یکپارچه‌سازی، و یک
سیستم‌عامل هوش مصنوعی با دروازه‌ی تأیید انسانی.

> **نکته‌ی نگهداری:** این فایل README تنها منبع معتبر برای «SOBI CRM چه
> کاری انجام می‌دهد» است. هرگاه قابلیت جدیدی به پروژه اضافه شد، باید اینجا
> (در هر سه فایل زبان) ثبت شود — نگذارید فهرست قابلیت‌ها قدیمی شود.

## شروع سریع

```bash
# ۱. اجرای Postgres + Mailpit
docker compose up -d

# ۲. تنظیم محیط
cp .env.example .env      # مقادیر توسعه از قبل تولید شده‌اند؛ در صورت نیاز تغییر دهید

# ۳. نصب، مایگریشن، seed
npm install
npx prisma migrate dev
npm run db:seed           # داده‌ی نمایشی CRM را به اولین فضای کاری شما متصل می‌کند

# ۴. اجرا
npm run dev               # http://localhost:3000
```

برای مشاهده‌ی لندینگ‌پیج بازاریابی به `/` بروید، یا مستقیم به
`/fa/register` بروید تا یک فضای کاری بسازید. در محیط توسعه، صفحه‌ی ورود
همچنین یک دکمه‌ی یک‌کلیکی **«ورود به فضای کاری دمو»** دارد (به بخش
[حالت دمو](#حالت-دمو) مراجعه کنید).

> **پورت‌ها:** Postgres در `docker-compose.yml` روی پورت **۵۴۳۳** نگاشت
> شده (در صورت تغییر، `DATABASE_URL` را هم تغییر دهید). Mailpit از
> ۱۰۲۵ (SMTP) / ۸۰۲۵ (وب) استفاده می‌کند.

## پشته‌ی فناوری

- **Next.js 16** (App Router، RSC) + TypeScript strict
- **PostgreSQL 16** + **Prisma 7** (اسکیمای چندفایلی، آداپتور درایور —
  `@prisma/adapter-pg`)
- **Better Auth** (ایمیل/گذرواژه، نشست‌های دیتابیسی) + یک لایه‌ی سفارشی
  مستأجر/RBAC
- **Tailwind CSS v4** + یک سیستم طراحی سفارشی مبتنی بر OKLCH (نه یک قالب
  آماده)
- جدول‌های داده به سبک TanStack، کانبان **dnd-kit**، گراف روابط
  **React Flow**، داشبوردهای **react-grid-layout**، **Recharts**،
  پالت فرمان **cmdk**
- بین‌المللی‌سازی با **next-intl** — انگلیسی، آلمانی و فارسی (RTL کامل،
  فونت Vazirmatn)
- تایپوگرافی **Manrope + DM Sans** برای سایت عمومی بازاریابی (Vazirmatn
  برای فارسی، هماهنگ با بقیه‌ی اپلیکیشن)

## فهرست قابلیت‌ها

### پلتفرم و چندمستأجری

- **چندمستأجری دفاع‌درعمق** — یک قابلیت fail-closed در پریزما context
  مستأجر تغییرناپذیر مبتنی بر AsyncLocalStorage را تزریق می‌کند، درحالی‌که
  RLS اجباری PostgreSQL همان مرز را مستقل از کد برنامه اعمال می‌کند.
  دسترسی‌های identity و کلاینت‌های system که در فهرست مجاز محدودی قرار
  دارند، از نقش‌های دیتابیسی جدا و حداقل‌دسترسی استفاده می‌کنند و هیچ
  escape hatch عمومی `rawDb` وجود ندارد.
- **راه‌اندازی مستأجر** — فضاهای کاری جدید از طریق یک فرایند onboarding
  هدایت‌شده ساخته می‌شوند که مستأجر، نقش Owner و پایپ‌لاین/مراحل پیش‌فرض را
  می‌سازد.
- **گذرگاه رویداد (Event Bus) + لاگ دائمی** — هر موتور رویدادهای تایپ‌شده
  را به یک باس درون‌فرایندی با جدول `Event` دائمی ارسال می‌کند؛ Timeline،
  Activity Feed، Analytics، اتوماسیون، اعلان‌ها، حسابرسی و یکپارچه‌سازی‌ها
  همگی به همان لاگ مشترک می‌پیوندند.
- **مدیریت قابلیت (Feature Management)** — فعال‌سازی ماژول و فلگ‌های
  بتا/آزمایشی به‌صورت رکوردهای `Feature` + `FeatureGrant` مدل‌سازی و به
  ازای هر درخواست ارزیابی می‌شوند.
- **هسته‌ی متادیتا** — موجودیت‌های داخلی (مخاطب، معامله، بیمه‌نامه، …)
  ساختار خود را در همان رجیستری متادیتای نسخه‌بندی‌شده‌ی JSONB ثبت می‌کنند
  که رکوردهای ساخته‌شده در زمان اجرا هم از آن استفاده می‌کنند.

### احراز هویت و دسترسی

- نشست‌های ایمیل/گذرواژه‌ی **Better Auth**، مبتنی بر دیتابیس، هم‌مبدأ
  (روی هر پورت/میزبانی بدون آدرس پایه‌ی هاردکد کار می‌کند).
- **RBAC** — نقش → مالکیت → دید تیم → قوانین محدودیت رکورد → override
  ادمین، از طریق `can()`/`authorize()`. کلیدهای دسترسی به‌صورت
  `"<ماژول>.<موجودیت>.<عملیات>"` با پشتیبانی از wildcard (`crm.*.read`، `*`).
- **نقش‌های سیستمی** به ازای هر مستأجر: مالک، مدیر سیستم، مدیر، کارمند،
  مشتری (پورتال).
- **قوانین سطح فیلد** و یک `AuditLog` کامل (دسته‌های احراز هویت، داده،
  فایل، دسترسی، خروجی، ادمین، امنیت، هوش مصنوعی) با یک نمایشگر در بخش
  مدیریت.
- **ورود دموی یک‌کلیکی** — به بخش [حالت دمو](#حالت-دمو) مراجعه کنید.

### مدیریت پلتفرم

- **راه‌اندازی خودکار سوپر ادمین** — اولین کاربری که در یک استقرار تازه
  ثبت‌نام می‌کند، به‌طور خودکار با فلگ `isSuperAdmin` علامت‌گذاری می‌شود؛
  نیازی به دستکاری دستی دیتابیس یا مرحله‌ی seed نیست. قابل واگذاری مجدد:
  اگر در حال حاضر سوپر ادمینی وجود نداشته باشد، ثبت‌نام بعدی سوپر ادمین
  می‌شود.
- **پنل `/platform-admin`** — یک سطح بین‌مستأجری (جدا از بخش مدیریت
  مخصوص هر مستأجر)، فقط برای سوپر ادمین قابل مشاهده، با ورودی اختصاصی
  خودش در Module Rail:
  - **پلن‌های قیمت‌گذاری** — CRUD کامل روی پلن‌های نمایش‌داده‌شده در بخش
    قیمت‌گذاری عمومی، به تفکیک هر زبان (نام، توضیحات، قیمت ماهانه/سالانه،
    متن دکمه، فهرست امکانات)، همراه با ترتیب و فلگ «پیشنهادی».
  - **محتوای لندینگ** — بازنویسی متن به تفکیک هر زبان برای مهم‌ترین
    متن‌های لندینگ (Hero، بنر CTA، توضیح قیمت‌گذاری)؛ فیلد خالی به متن
    پیش‌فرض برمی‌گردد — بدون نیاز به استقرار مجدد برای یک تغییر سریع متن.
  - **برندینگ** — لوگو/فاویکون به‌صورت آدرس تصویر میزبانی‌شده در جای دیگر
    (نه آپلود فایل — سیستم فایل سرورلس Vercel در زمان اجرا فقط‌خواندنی
    است)، اعمال‌شده در همه‌جایی که آرم SOBI CRM نمایش داده می‌شود: نوار
    فضای کاری، پنل ورود/ثبت‌نام، صفحه‌ی عمومی قرارداد و لندینگ‌پیج/فوتر.
  - **نوار اعلان** — یک نوار تبلیغاتی/اطلاعیه (کد تخفیف، اعلامیه‌ها) بالای
    لندینگ‌پیج عمومی و فضای کاری داخل برنامه، با متن به تفکیک هر زبان،
    رنگ پس‌زمینه/متن، و یک انیمیشن اسکرول (چپ↔راست یا ثابت) صرفاً با CSS.
  - محافظت‌شده توسط `requireSuperAdmin()`، یک فلگ سراسری پلتفرم، جدا از
    بررسی‌های دسترسی `can()`/`authorize()` مخصوص هر مستأجر که در باقی
    محصول استفاده می‌شود.
- **ادیتور متن حرفه‌ای** — یک ادیتور کامل مبتنی بر TipTap (بولد/ایتالیک/
  لیست/لینک) پشتوانه‌ی هر فیلد محتوایی است که سوپر ادمین می‌نویسد (بازنویسی
  محتوای لندینگ، پنل ویرایش با موس که در ادامه می‌آید)؛ HTML ذخیره‌شده
  پیش از persist یا رندر، سمت سرور پاک‌سازی می‌شود (`isomorphic-dompurify`).
- **حالت CMS زنده با ویرایش روی هاور** — یک گزینه‌ی «بازگشت به سایت اصلی»
  در پایین Module Rail (فقط سوپر ادمین) لندینگ‌پیج عمومی را درحالی‌که هنوز
  لاگین هستید باز می‌کند. آنجا، بردن موس روی هر متن قابل‌ویرایش (Hero،
  بنر CTA، توضیح قیمت‌گذاری) یک مداد نمایش می‌دهد که ادیتور متن حرفه‌ای را
  دقیقاً همان‌جا باز می‌کند — بدون نیاز به پیدا کردن فیلد مربوطه در پنل ادمین.

### هسته‌ی CRM

- **مخاطبین و شرکت‌ها** — CRUD کامل، find-or-create شرکت با تطبیق ایمن در
  برابر LIKE-injection، روابط مخاطب↔شرکت، برچسب‌ها، یادداشت‌ها. فهرست،
  کسب‌وکار/خدمت (نام شرکت، یا یادداشت علاقه‌مندی خدمت برای مخاطبین
  انفرادی)، تلفن، ایمیل، تاریخ ثبت‌نام، مرحله‌ی چرخه‌ی عمر و منبع سرنخ
  (وب‌سایت، دستی، چت‌بات، تلگرام، وب‌چت) را نشان می‌دهد. تایم‌لاین صفحه‌ی
  جزئیات ۳۶۰ درجه یک تاریخچه‌ی واقعاً سرتاسری است — ادغام‌شده از فعالیت خود
  مخاطب **و فعالیت سرنخ اولیه‌اش** — به‌طوری‌که از اولین لحظه‌ی ورودی تا
  تبدیل و هر آنچه پس از آن رخ داده قابل‌خواندن است، به‌همراه یک کنترل
  دستی «افزودن فعالیت» (ثبت یک تماس/جلسه، یا ساخت یک وظیفه‌ی واقعی و
  متصل به‌عنوان یادآور پیگیری، که در فضای کاری عملیات ← وظایف هم قابل
  مشاهده است). شرکت‌ها هم همان رفتار را دریافت می‌کنند — یک کارت جزئیات
  قابل‌ویرایش و همان کنترل «افزودن فعالیت» — به‌علاوه یک **تایم‌لاین
  تجمیعی**: فعالیتی که روی هر یک از مخاطبین آن شرکت ثبت می‌شود، آنجا هم
  ظاهر می‌شود، با برچسب نام همان مخاطب، به‌طوری‌که رابطه در سراسر هر فرد
  آن کسب‌وکار همگام می‌ماند.
- **سرنخ‌ها** — دریافت از فرم وب‌سایت و چت‌بات در یک صف مشترک؛ هر سرنخ
  `source`، `industry` (حوزه‌ی فعالیت)، `conversationId` (اتصال به گفتگوی
  چت‌بات) و `customFields` آزاد دارد. یک فهرست کارت‌محور با فیلتر وضعیت،
  جستجو و خروجی CSV؛ یک نقطه‌ی ورود دستی «سرنخ جدید» در کنار فرم عمومی
  وب‌سایت؛ و یک صفحه‌ی جزئیات کامل (`/crm/leads/[id]`) با یک کارت جزئیات
  قابل‌ویرایش، امتیازدهی هوشمند، یک پیشنهاد محتوای هوشمند و دیالوگ تبدیل
  غنی‌تر که در ادامه توضیح داده می‌شود.
- **معاملات و پایپ‌لاین** — یک موتور رکورد مرحله‌ای عمومی (بورد کانبان،
  Drag-and-Drop بومی) که هم توسط معاملات و هم پنج پایپ‌لاین ماژول صنعتی
  استفاده می‌شود؛ مراحل قابل‌تنظیم با فلگ‌های `isWon`/`isLost` که وضعیت
  معامله را خودکار می‌کنند. پایپ‌لاین پیش‌فرض معامله پنج مرحله‌ی فعال دارد
  — جدید، در حال بررسی، مشاوره، پیشنهاد ارسال‌شده، مرحله‌ی نهایی قرارداد —
  به‌علاوه‌ی دو نتیجه‌ی نهایی برد/باخت. هر کارت همچنین یک منوی «انتقال به
  هر مرحله» به‌عنوان جایگزینی مناسب لمس در برابر Drag-and-Drop دارد؛ زیر
  بورد، یک نوار خلاصه‌ی معاملات بسته/در انتظار، ارزش و تعداد معاملات
  برنده، بازنده و هنوز باز را جمع می‌زند.
- **فعالیت‌ها و یادداشت‌ها** — یک تایم‌لاین برای هر رکورد (تماس، ایمیل،
  جلسه، یادداشت، تغییر مرحله، فایل، رویداد سیستمی)، خودکار از هر چیزی که
  رویداد منتشر می‌کند تغذیه می‌شود.
- **Activity Feed سراسری** (`/crm/activity`) — هر رویداد معنادار پلتفرم در
  یک جریان زمانی و قابل‌فیلتر واحد: فیلترهای نوع موجودیت و بازه‌ی زمانی،
  هر ورودی نام واقعی عامل را نشان می‌دهد و مستقیماً به رکورد مربوطه لینک
  می‌شود، با برچسب‌های کاملاً بومی‌سازی‌شده برای همه‌ی ~۴۵ نوع رویداد
  (en/de/fa). یک اقدام «ثبت فعالیت» به کاربر اجازه می‌دهد یک تماس، جلسه،
  ایمیل یا یادداشت را برای هر مخاطب/شرکت/معامله/سرنخی که از طریق یک
  انتخابگر type-ahead پیدا شده ثبت کند — همان ورودی روی تایم‌لاین خودِ آن
  رکورد هم ظاهر می‌شود.
- **برچسب‌ها و روابط** — یک مدل عمومی `Relationship` به‌همراه یک
  **گراف روابط** مستقل (`/crm/graph`، React Flow): هر شرکت، مخاطب و معامله
  به‌صورت یک بلوک رنگی و قابل‌جابه‌جایی نمایش داده می‌شود (آیکون + عنوان +
  یک خط واقعیت — شرکت تعداد مخاطبان/معاملاتش را نشان می‌دهد، مخاطب نام
  شرکتش را، معامله مبلغش را)، به‌صورت خودکار به شکل درختی از چپ به راست
  چیده شده تا شکل کلی یک حساب در یک نگاه خوانده شود. بلوک‌ها آزادانه
  جابه‌جا می‌شوند؛ کلیک روی یک بلوک ارتباط‌هایش را فوکوس می‌کند (همسایه‌ها و
  ارتباط‌هایشان روشن می‌شوند، بقیه محو می‌شوند)؛ تراشه‌های فیلتر نوع،
  شرکت‌ها/مخاطبان/معاملات را نمایش/مخفی می‌کنند؛ یک مینی‌مپ و pan/zoom هم
  برای گراف‌های بزرگ‌تر کمک می‌کنند. فراتر از ارتباط‌هایی که کلیدهای خارجی
  موجود تعیین می‌کنند، کاربر می‌تواند **با کشیدن از یک بلوک به بلوک دیگر
  یک ارتباط جدید بسازد** — این ارتباط به‌صورت یک ردیف واقعی `Relationship`
  ذخیره می‌شود (نه فقط یک طرح بصری موقت) و از همان بوم قابل حذف هم هست؛
  یال‌های خودکارِ مشتق‌شده (که یک کلید خارجی واقعی در جای دیگر را منعکس
  می‌کنند) عمداً از اینجا قابل حذف نیستند.
- **درون‌ریزی/برون‌ریزی CSV** برای موجودیت‌های اصلی.

### تبدیل سرنخ و امتیازدهی هوشمند

- **گردش‌کار تبدیل آگاهانه** (`convertLead`) — یک سرنخ خام را به مخاطب
  تبدیل می‌کند، شرکت آن را find-or-create می‌کند، در صورت نیاز یک معامله
  می‌سازد، و پیام اصلی سرنخ را به‌عنوان اولین یادداشت تایم‌لاین ثبت می‌کند.
  Idempotent است: تبدیل مجدد، همان مخاطب موجود را برمی‌گرداند نه یک کپی.
  دیالوگ تبدیل فیلدهای غنی‌تری از آنچه سرنخ خام دارد ثبت می‌کند — نام/
  نام‌خانوادگی، عنوان شغلی، ایمیل/تلفن/شرکت اصلاح‌شده، یک حوزه‌ی فعالیت،
  یک علاقه‌مندی خدمت برای سرنخ‌های انفرادی (غیرتجاری)، و یک یادداشت
  اضافی — بدون این‌که هرگز خود سرنخ را بازنویسی کند.
- **امتیازدهی هوشمند سرنخ** — امتیازی از ۰ تا ۱۰۰ همراه با یک توضیح.
  در نبود کلید API هوش مصنوعی، به یک **هیوریستیک** شفاف و قابل‌توضیح
  (کامل‌بودن ایمیل/تلفن/شرکت/منبع/پیام) سقوط می‌کند تا این قابلیت همیشه
  قابل نمایش باشد.
- **خلاصه‌سازی گفتگو** — `Conversation` چت‌بات متصل به یک مخاطب را پیدا
  کرده و آن را در ۳ تا ۵ بولت روی کارت «شناخت مشتری» خلاصه می‌کند.

### سیستم‌عامل هوش مصنوعی

مسیر: `ارائه‌دهنده‌ها ← کتابخانه‌ی پرامپت ← Skillها ← ابزارها ← حلقه‌ی
عامل ← Action Center ← تأیید انسانی ← حسابرسی هوش مصنوعی`.

- **ارائه‌دهنده‌ها** — OpenAI، OpenRouter و یک اندپوینت محلی سازگار با
  OpenAI، قابل انتخاب برای هر مستأجر. بدون کلید تنظیم‌شده، یک
  **ارائه‌دهنده‌ی mock** بدون کلید، برای هر قابلیت هوش مصنوعی خروجی
  هیوریستیک مفید تولید می‌کند — چیزی هرگز کرش نمی‌کند یا به دلیل نبود
  کلید API بی‌صدا کاری نمی‌کند.
- **Skillها** — خلاصه‌ی رکورد، پیشنهاد قدم بعدی (به‌عنوان یک اقدام
  در‌انتظار)، پیش‌نویس ایمیل، تشخیص مدرک ناقص (قانون‌محور، بدون LLM)،
  امتیازدهی سرنخ، خلاصه‌سازی گفتگو و پیشنهاد محتوا (به بخش
  [پایگاه دانش](#پایگاه-دانش-و-پیشنهاد-محتوای-هوشمند) مراجعه کنید).
- **دستیار ابزارفراخوان («چت با CRM»)** — یک حلقه‌ی عامل محدود (حداکثر
  ۴ دور) با چهار ابزار فقط‌خواندنی و اعتبارسنجی‌شده با zod —
  `query_leads`، `query_deals`، `query_activities`، `crm_stats` — مستقیماً
  از دیتابیس تغذیه می‌شوند. system prompt الزام می‌کند که مدل به‌جای
  ساختن یک عدد، یک ابزار را فراخوانی کند؛ ارائه‌دهنده‌ی mock این را به‌شکل
  مکانیکی با بازتاب‌دادن فقط JSON واقعی ابزارها اجرا می‌کند. پاسخ‌ها
  کلمه‌به‌کلمه به کلاینت stream می‌شوند.
- **Action Center** — قابلیت‌های هوش مصنوعی که قرار است داده بنویسند
  هرگز مستقیم این کار را نمی‌کنند؛ یک `AiAction` در‌انتظار می‌سازند که
  یک انسان صریحاً آن را **تأیید** یا **رد** می‌کند. تأییدها رویداد
  `ai.action_approved` منتشر می‌کنند و تحت دسته‌ی هوش مصنوعی حسابرسی
  می‌شوند. تولید ایمیل کمپین هم از همین اصل انسان-در-حلقه پیروی می‌کند.
- **حسابرسی هوش مصنوعی** — هر فراخوانی هوش مصنوعی در `AiLog` ثبت می‌شود.

### قراردادها

- قراردادهای **شماره‌گذاری‌شده‌ی خودکار** (`CTR-<سال شمسی>-<توالی>`)
  تولیدشده از یک معامله/مخاطب/شرکت، متصل به نام همان مخاطب مشخص در فهرست
  قراردادها. **چهار قالب متنی قابل‌انتخاب** (خدمات مشاوره، توسعه‌ی
  نرم‌افزار، حق‌الزحمه‌ی ماهانه، پروژه‌ی یک‌باره)، هرکدام یک سند حقوقی
  کامل به زبان فارسی با مقادیر واقعی جایگزین‌شده.
- **طراح سربرگ و امضای دیجیتال** (در همان ماژول، سراسر مستأجر): نام/لوگو/
  آدرس/فوتر شرکت، نام + عنوان امضاکننده، و یک تقویم **شمسی/میلادی**
  قابل‌تنظیم توسط ادمین برای اسناد.
- **الزام امضای دیجیتال پیش از اشتراک‌گذاری** — یک قرارداد باید پیش از
  ارسال، دیجیتالی امضا شود (`applySignature`)؛ در غیر این صورت
  `sendContract` از صدور لینک اشتراک‌گذاری خودداری می‌کند. بلوک امضا
  (نام/عنوان امضاکننده، تاریخ امضا و یک کد QR تولیدشده به‌صورت محلی —
  هرگز به یک سرویس QR شخص ثالث ارسال نمی‌شود، چون توکن غیرقابل‌حدس
  اشتراک‌گذاری را کدگذاری می‌کند) در انتهای هر سند امضاشده، هم در صفحه‌ی
  عمومی و هم در PDF قابل‌دانلود، نمایش داده می‌شود.
- **خروجی پیش‌قرارداد** — یک پیش‌نمایش PDF واترمارک‌دار («پیش‌قرارداد» +
  سربرگ) برای بررسی غیررسمی مشتری از مرحله‌ی پیش‌نویس، تولیدشده از طریق
  `/crm/contracts/[id]/print?mode=pre`؛ هرگز شامل امضای دیجیتال نمی‌شود.
  PDF نهاییِ امضاشده (`?mode=final`) پس از امضا در دسترس است. هر دو مسیر
  از چاپ-به-PDF بومی مرورگر استفاده می‌کنند (بدون وابستگی به رندر سمت
  سرور) با یک stylesheet چاپ اختصاصی.
- **صفحه‌ی اشتراک‌گذاری عمومی** — یک لینک `shareToken` غیرقابل‌حدس که در
  آن مشتری سند برند‌شده با سربرگ را بررسی می‌کند، بلوک امضا+QR را می‌بیند
  و با واردکردن نام خودش به‌همراه شماره‌ی قراردادی که در صفحه نشان داده
  شده (یک بررسی متقابل سبک) **آنلاین می‌پذیرد**؛ اسکن QR همان صفحه را باز
  می‌کند. ردیابی بازدید (`sent → viewed`) و پذیرش (`viewed → accepted`)
  در سمت سرور await می‌شوند (نه fire-and-forget) تا قطع پاسخ سرورلس نتواند
  نوشتن را از دست بدهد.
- **چرخه‌ی وضعیت**: پیش‌نویس ← امضای دیجیتال‌شده ← ارسال‌شده ← بازدیدشده ←
  پذیرفته‌شده/لغوشده، با قفل ویرایش پس از پذیرش.
- **بازنویسی هوشمند** متن قرارداد و تولید پیام **پیگیری هوشمند**.
- انتشار رویداد در context عمومی
  (`contract.created|sent|viewed|accepted`) حتی بدون نشست احرازشده هم کار
  می‌کند، از طریق یک helper به نام `publicContext(tenantId)` که نوشتن را
  در یک `PlatformContext` حداقلی بسته‌بندی می‌کند — به‌طوری‌که اتوماسیون/
  webhookها همچنان برای این رویدادها اجرا می‌شوند.

### کمپین‌های ایمیلی

- **ماژول سازنده‌ی سگمنت** (`engines/campaigns/segments.ts`)، جدا از موتور
  کمپین — resolverهای مخاطب کدمحور و نام‌گذاری‌شده (سرنخ‌های ازدست‌رفته،
  سرنخ‌های بدون پیگیری، معاملات ازدست‌رفته، مشتریان برنده)، هرکدام محدود
  به ۲۰ گیرنده در هر اجرای کمپین. هر سگمنت همچنین **آمار زنده‌ی** بدون سقف
  را (اندازه‌ی واقعی کل، چند نفر از طریق ایمیل قابل‌دسترسی هستند، و — برای
  سگمنت‌های مبتنی بر معامله — ارزش کل) در انتخابگر ساخت کمپین، پیش از
  نهایی‌کردن، نشان می‌دهد.
- **شخصی‌سازی هوشمند به ازای هر گیرنده** — یک درخواست در هر لحظه (هرگز
  دسته‌ای/موازی)، یک system prompt با حداکثر ۱۲۰ کلمه/بدون فروش سخت/CTA
  مشاوره‌ی رایگان، با fallback به mock در نبود کلید هوش مصنوعی. هم به‌صورت
  تک‌گیرنده و هم به‌صورت **«تولید همه»ی گروهی** کار می‌کند (و، پس از تأیید
  پیش‌نویس‌ها، یک **«ارسال همه‌ی آماده‌ها»ی گروهی**) — هر دو همچنان کاملاً
  ترتیبی، به‌طوری‌که یک کمپین هرگز موجی از درخواست‌های موازی به ارائه‌دهنده‌ی
  هوش مصنوعی یا ارائه‌دهنده‌ی ایمیل ارسال نمی‌کند.
- **بازبینی انسان-در-حلقه** — هر ایمیل تولیدشده قابل‌ویرایش است، می‌تواند
  دوباره تولید یا رد شود، و فقط پس از تأیید ارسال می‌شود (تکی، یا از طریق
  اقدام گروهی «ارسال همه‌ی آماده‌ها» پس از بازبینی). خود کمپین به‌محض
  رسیدن هر گیرنده به یک وضعیت نهایی به‌طور خودکار کامل می‌شود
  (`status: "sent"`)، و نمای فهرست یک شمارنده‌ی زنده‌ی ارسال‌شده/کل به ازای
  هر کمپین نشان می‌دهد، نه فقط تعداد خام گیرندگان.
- **ارائه‌دهنده‌ی ایمیل قابل‌انتخاب** — SMTP (پیش‌فرض، Mailpit در توسعه)،
  **Resend** یا **Amazon SES** از طریق `EMAIL_PROVIDER`، به‌طوری‌که ارسال
  گروهی در production می‌تواند بدون تغییر کد به یک ارائه‌دهنده‌ی واقعی
  ایمیل تراکنشی اشاره کند.
- **حسابداری دقیق تحویل** — ارسال کمپین از `emailChannel.sendStrict`
  استفاده می‌کند (خطاهای واقعی ارسال را دوباره پرتاب می‌کند تا یک ارسال
  به‌عنوان `failed` ثبت شود، نه اینکه بی‌صدا «موفق» تلقی شود)، متفاوت از
  `emailChannel.send` با تلاش-بهترین که برای پخش اعلان درون‌برنامه‌ای
  استفاده می‌شود.
- **Webhook ایمیل ورودی** (`POST /api/v1/inbound-email`) — یک دریافت‌کننده
  مستقل از ارائه‌دهنده (SendGrid Inbound Parse، Mailgun Routes، Postmark
  و مشابه، همگی شکلی مشابه را forward می‌کنند) برای پاسخ‌ها: یک secret
  مشترک را بررسی می‌کند، مستأجر را از آدرس «به» resolve می‌کند و پردازش
  واقعی را به‌عنوان یک job پس‌زمینه در صف قرار می‌دهد، به‌طوری‌که
  ارائه‌دهنده فقط منتظر یک 200 OK سریع می‌ماند، نه یک نوشتن دیتابیس. این
  job پیام را در همان تاریخچه‌ی `Communication` که ارسال‌های خروجی از قبل
  استفاده می‌کنند ثبت می‌کند (تطبیق‌داده‌شده با یک مخاطب/سرنخ بر اساس آدرس
  فرستنده)، و آن را روی تایم‌لاین آن رکورد و Activity Feed سراسری نمایش
  می‌دهد.

### پایگاه دانش و پیشنهاد محتوای هوشمند

- یک **پایگاه دانش** داخلی سبک (`KnowledgeArticle`: عنوان، متن، برچسب‌ها)
  که از فضای کاری CRM قابل ویرایش است.
- **Skill پیشنهاد محتوای هوشمند** — برای یک سرنخ، هر مقاله را بر اساس
  ارتباط کلیدواژه/برچسب امتیاز می‌دهد (هرگز مقاله‌ای نمی‌سازد)، بهترین
  تطابق را انتخاب می‌کند و یک پیام پیگیری کوتاه و مبتنی بر واقعیت می‌نویسد
  که فقط به محتوای واقعی همان مقاله ارجاع می‌دهد. اگر سرنخ قبلاً تبدیل شده
  باشد، پیام همچنین به‌عنوان یادداشت روی تایم‌لاین مخاطب مرتبط ثبت می‌شود.

### گزارش‌ها و بینش‌ها

- **نمودارها به‌عنوان حالت پیش‌فرض** (`/crm/reports`) — چهار نمودار
  Recharts ابتدا نمایش داده می‌شوند تا کسب‌وکار در یک نگاه ببیند واقعاً چه
  کاری انجام داده: یک ردیف KPI (کل سرنخ‌ها، نرخ تبدیل سرنخ به برد،
  پایپ‌لاین باز، درآمد ۱۲ ماهه)، یک قیف تبدیل (سرنخ ← تبدیل‌شده ← معامله ←
  گذشته از مرحله‌ی اول ← برد، همراه با درصد نسبت به بالای قیف)، **ارزش
  پایپ‌لاین به تفکیک مرحله** (هم‌رنگ با tone هر مرحله، به‌طوری‌که یک مرحله
  اینجا همان رنگی را نشان می‌دهد که روی بورد معاملات دارد)، تفکیک منابع
  سرنخ، و درآمد ۱۲ ماهه با **تقویم شمسی** (سطل‌های واقعی ماه شمسی از طریق
  `jalaali-js`، نه ماه‌های میلادی بازبرچسب‌گذاری‌شده). نمودارها صرف‌نظر از
  زبان فعال، درون یک ناحیه‌ی رسم اجباراً LTR رندر می‌شوند تا هندسه‌ی
  محور/میله هرگز زیر راست‌چین بهم نریزد، درحالی‌که برچسب‌های تیک همچنان در
  جهت خواندنِ زبان فعال نمایش داده می‌شوند.
- **گزارش‌های جدولی** (معاملات، پایپ‌لاین، وظایف، مخاطبین) با خروجی CSV،
  حسابرسی‌شده — برای هرکس که داده‌ی خام را به‌جای خلاصه‌ی تصویری بخواهد، با
  یک کلیک پشت یک **کلید تعویض نمودارها/جدول‌ها**.
- هر برچسب روی این صفحه — گام‌های قیف، نام مراحل پایپ‌لاین، منابع سرنخ —
  یک کلید ترجمه است که در en/de/fa resolve می‌شود، نه یک رشته‌ی خام
  داخل موتور آنالیتیکس؛ همان نام‌های مرحله اکنون دقیقاً بین بورد معاملات،
  ویجت پایپ‌لاین داشبورد CRM، ویجت پایپ‌لاین سازنده‌ی داشبورد قابل‌تنظیم و
  این نمودار مطابقت دارند.

### موتور فرم‌ساز

- **سازنده‌ی فرم** با Drag-and-Drop: بخش‌ها/تب‌ها، نمایش شرطی و فیلدهای
  محاسبه‌شده (از طریق ارزیاب عبارت مشترک Business Rules)، اعتبارسنجی
  تولیدشده از متادیتا به Zod، برچسب‌های چندزبانه، تکرارشونده‌های ساده،
  نسخه‌بندی، قالب‌های قابل‌استفاده‌ی مجدد.

### فایل‌ها، وظایف، تقویم، اعلان‌ها

- **نمای کلی عملیات** (`/ops`) — یک صفحه‌ی ورودیِ فضای کاری که تعداد وظایف
  باز/عقب‌افتاده، رویدادهای تقویم پیش‌رو و تعداد فایل‌های ذخیره‌شده را در
  یک نگاه جمع می‌زند، با میان‌برهای سریع به وظایف، تقویم و فایل‌ها.
- **موتور فایل/سند** — آپلود امن، دسته‌بندی‌ها، نسخه‌ها، پیش‌نمایش،
  انقضا، چک‌لیست مدارک الزامی به ازای هر رکورد.
- **وظایف** — زیروظیفه، تکرار، وابستگی، نظرات، تشخیص تأخیر از طریق یک job
  زمان‌بندی‌شده.
- **تقویم** — نمای ماهانه/هفتگی، در‌دسترس‌بودن، تشخیص تداخل رزرو (مشترک با
  ماژول‌های صنعتی خدماتی).
- **اعلان‌ها** — مرکز درون‌برنامه‌ای، کانال ایمیل، ترجیحات کاربر،
  زمان‌بند یادآوری/تأخیر.
- **تاریخچه‌ی ارتباطات** — یک لاگ یکپارچه به ازای هر رکورد برای ایمیل‌ها/
  تماس‌ها در همه‌ی کانال‌ها.

### داشبوردها

- **سازنده‌ی داشبورد** — یک بوم `react-grid-layout` با کتابخانه‌ی ویجت
  (KPI، تفکیک پایپ‌لاین، روند فعالیت، وظایف، Activity Feed، ویجت‌های
  ثبت‌شده توسط ماژول)، چیدمان‌های شخصی/نقشی/مستأجری/مشترک، و قالب‌های
  نسخه‌بندی‌شده.

### مالی

- **فضای کاری مالی** — جمع‌بندی درآمد بین‌ماژولی: درآمد برنده‌شده و ارزش خط
  لوله‌ی باز از معاملات، ارزش کل قراردادها به‌تفکیک وضعیت، نرخ برد، معاملات
  اخیراً برنده‌شده، و وضعیت صورت‌حساب/اشتراک مستأجر.

### سازنده‌ی گردش‌کار و اتوماسیون

- **سازنده‌ی گردش‌کار بصری** — مراحل/گام‌ها، زنجیره‌ی تأیید هدایت‌شده با
  Rules، شرایط، تایمر، SLA، تشدید، فیلد/مدرک الزامی، قالب‌ها، نسخه‌بندی؛
  هر فیلد، هر گزینه‌ی رنگ مرحله و هر کنترل در هر سه زبان ترجمه شده است
  (این صفحه در نسخه‌ای پیش‌تر کاملاً انگلیسی بود).
- **موتور اتوماسیون** — trigger رویداد/زمان‌بندی ← شرایط Rules ← اقدامات
  (ساخت وظیفه، اعلان، به‌روزرسانی فیلد، جابه‌جایی مرحله، ارسال ایمیل،
  فراخوانی webhook، اجرای Skill هوش مصنوعی)، با لاگ کامل `AutomationRun`
  که به Observability می‌رود.
- یک **ارزیاب عبارت JSON-AST** مشترک و sandbox‌شده، منطق شرطی فرم، gateهای
  گردش‌کار و شرایط اتوماسیون را تغذیه می‌کند — یک پیاده‌سازی، سه مصرف‌کننده.

### یکپارچه‌سازی‌ها

- **وبهوک‌های خروجی** — امضاشده (`HMAC-SHA256`، هدرهای
  `X-Sobi-Signature` / `X-Sobi-Event` / `X-Sobi-Delivery` پایدار)،
  اشتراک به تفکیک مستأجر بر اساس نوع رویداد، تحویل دائمی و مستقل
  تکرارشونده با ثبت وضعیت، اجبار HTTPS، مسدودسازی شبکه‌های خصوصی/metadata،
  pin کردن DNS و عدم دنبال‌کردن redirect.
- **کلیدهای API دارای scope** برای REST API عمومی (`/api/v1/...`) با
  شناسه‌ی throttle هش‌شده، صفحه‌بندی cursor، پاسخ‌های خطای پایدار، سهمیه‌ی
  ماهانه، یک قرارداد OpenAPI و یک SDK تایپ‌اسکریپت بدون وابستگی.
- **هسته‌ی تجاری SaaS** — subscription مستقل از ارائه‌دهنده، دوره‌ی
  آزمایشی، entitlement/limit پلن قابل‌خواندن ماشینی، شمارنده‌های مصرف
  اتمیک، و یک fallback محافظه‌کارانه برای پلن رایگان. صورت‌حساب دستی صریح
  است تا زمانی‌که یک آداپتور PSP پیکربندی شود.
- **درون‌ریزی CSV مخاطبین** — ذخیره‌ی خصوصی منبع، تجزیه و mapping محدود،
  پردازش پایدار، خلاصه‌ی خطا به ازای هر ردیف، بررسی سهمیه و مختصات
  retry ایدمپوتنت.
- زیرساخت OAuth و ارائه‌دهندگان شخص ثالث (Google، Microsoft، WhatsApp،
  Telegram، Stripe، PayPal) طراحی شده اما به اعتبارنامه‌ی واقعی متصل نشده
  — به [`docs/ROADMAP.md`](docs/ROADMAP.md) مراجعه کنید.

### ماژول‌های صنعتی

هر ماژول یک ترکیب سبک از موتورهای مشترک (Pipeline، Booking، Documents،
Finance) به‌همراه یک manifest (فضای کاری، ناوبری، دسترسی‌ها، انواع
رابطه، i18n) است. **هر هشت ماژول یک داشبورد اختصاصی (کارت‌های KPI + یک
فهرست موارد پیش‌رو/اخیر) و نمای فهرست با گردش‌کار ساخت** را ارائه
می‌دهند، همراه با داده‌ی نمایشی از پیش پر شده:

| ماژول | ترکیب |
|---|---|
| **بیمه** | محصولات، بیمه‌گرها، بیمه‌نامه‌ها با یادآوری تمدید، خسارات، کمیسیون‌ها |
| **وام و بانکداری** | درخواست‌ها، شرکای بانکی، پروفایل رمزنگاری‌شده‌ی متقاضی، چک‌لیست اعتباری، برنامه‌های بازپرداخت |
| **املاک** | ملک‌ها، نقش خریدار/فروشنده، بازدیدها، پیشنهادها، قراردادها، تطبیق مشتری↔ملک |
| **فروش و آژانس** | کمپین‌ها، پیشنهادها، اهداف، عملکرد |
| **مهاجرت** | پرونده‌های ویزا/مجوز، ارسال به مراجع، قالب مدارک الزامی، مهلت‌ها، بسته‌های خدماتی |
| **آرایشگاه مردانه** | خدمات، کارکنان، صندلی‌ها، نوبت‌ها، مراجعه‌ی حضوری، تاریخچه‌ی بازدید |
| **سالن زیبایی** | + دوره‌های درمانی، عکس قبل/بعد، فرم رضایت |
| **رستوران** | رزرو میز، حساسیت/ترجیحات مهمان، سرنخ‌های پذیرایی/کترینگ، وفاداری |

شش ماژول اسپک اضافی (سرمایه‌گذاری، حقوقی، آموزش، سلامت، خدمات و نگهداری،
مدیریت پروژه) به‌عنوان scaffold «به‌زودی» در فعال‌سازی ماژول ثبت شده‌اند.

هر داشبورد، فهرست و دیالوگ در هر هشت ماژول — به‌همراه کامپوننت‌های نوبت‌دهی
مشترک آرایشگاه مردانه، سالن زیبایی و رستوران — به‌طور کامل به انگلیسی،
آلمانی و فارسی ترجمه شده است؛ enumهای وضعیت/نوع (وضعیت بیمه‌نامه، هدف وام،
نوع ملک، نوع ویزا، وضعیت نوبت، …) به‌جای نمایش کد خام، برای هر مقدار به یک
کلید ترجمه‌ی امن نگاشت شده‌اند.

### استودیوی کم‌کد

- **قالب‌های صنعتی** — یک راهکار آماده‌ی صنعتی (باشگاه بدنسازی، مطب
  دندان‌پزشکی، شرکت مشاوره‌ای، …) را اعمال کنید که موجودیت‌های پیکربندی‌شده
  و داده‌ی نمونه را روی هسته‌ی کم‌کد نمونه‌سازی می‌کند. صنایع *داده‌اند،
  نه کد*: یک کسب‌وکار صنعت خود را فعال می‌کند و به‌جای یک ماژول کدنویسی‌شده،
  یک نقطه‌ی شروع قابل‌سفارشی‌سازی می‌گیرد.
- **سازنده‌ی موجودیت** — ساخت یک موجودیت سفارشی (فیلدها، روابط، دسترسی‌ها)
  که یک API CRUD تولیدشده، نمای لیست/جزئیات، جستجو و اتصال تایم‌لاین
  دریافت می‌کند.
- **قوانین کسب‌وکار** — ویرایشگری اختصاصی برای قوانین قابل‌استفاده‌ی مجدد
  اعتبارسنجی، واجد شرایط بودن، تأیید و محاسبه، که به‌صورت AST امنِ عبارت ذخیره و
  در فرم‌ها، گردش‌کارها و اتوماسیون‌ها ارزیابی می‌شوند.
- **قالب‌ها** — ساخت قالب‌های قابل‌استفاده‌ی مجدد برای ایمیل، سند، اعلان، گزارش
  و پرامپت هوش مصنوعی با درج متغیر `{{variable}}` و تشخیص زنده‌ی متغیرها.
- پنل‌های **تاریخچه‌ی نسخه**، مشترک بین همه‌ی سازنده‌ها.
- **پلتفرم فرمان** — یک پالت فرمان `Ctrl+K`.
- **جستجوی جامع** — جستجوی تمام‌متن Postgres در موجودیت‌ها، فایل‌ها،
  یادداشت‌ها، فعالیت‌ها، گزارش‌ها و فرمان‌ها.

### امنیت، حسابرسی و GDPR

- هش‌کردن گذرواژه با Argon2، نشست‌های دیتابیسی (httpOnly، SameSite)، در
  production محدودسازی نرخ توزیع‌شده مبتنی بر Redis.
- CSP با nonce مجزای هر درخواست (`strict-dynamic` و بدون inline script در
  production)، HSTS، `X-Content-Type-Options`، `X-Frame-Options`،
  `Referrer-Policy` و هدرهای امنیتی `Permissions-Policy`.
- رمزنگاری سطح فیلد AES-256-GCM برای داده‌ی حساس.
- Soft delete + سطل زباله برای موجودیت‌های اصلی؛ ردپای حسابرسی کامل.
- GDPR: رکورد رضایت، بسته‌ی خروجی داده‌ی شخصی
  (`/api/v1/gdpr/contacts/:id/export`)، گردش‌کار حذف با ناشناس‌سازی +
  سیاست‌های نگهداری قابل‌تنظیم.
- لاگ‌گیری ساختاریافته (Pino) به‌همراه داشبورد سلامت مدیریتی.

### لندینگ‌پیج بازاریابی

- یک صفحه‌ی عمومی بازاریابی کامل در `/` (فقط برای بازدیدکننده‌ی
  واردنشده — یک نشست احرازشده همچنان مستقیم به `/crm` هدایت می‌شود)،
  از نظر بصری مستقل از سیستم طراحی درون‌برنامه‌ای (پالت پاین/تیل و
  تایپوگرافی Manrope/DM Sans مخصوص خودش).
- بخش‌ها: نوار چسبان، Hero (با یک ماکت زنده‌ی جدول سرنخ‌ها، نه یک
  اسکرین‌شات)، یک بخش تیره‌ی مشکل/راه‌حل مبتنی بر گردش واقعی تبدیل سرنخ،
  یک توضیح گردش‌کار چهارمرحله‌ای، یک پیش‌نمایش زنده‌ی آنالیتیکس، تب‌های
  راهکار به ازای هر صنعت، قیمت‌گذاری نمایشی (صریحاً به‌عنوان «پیشنهاد
  تجاری واقعی نیست» علامت‌گذاری‌شده)، یک بخش سوالات متداول و یک بنر CTA
  دمو/ثبت‌نام.
- کاملاً بومی‌سازی‌شده (انگلیسی/آلمانی/فارسی، RTL برای فارسی).
- دکمه‌های «امتحان دمو» / «ورود به فضای کاری دمو» همان ورود دموی
  یک‌کلیکی صفحه‌ی ورود را فعال می‌کنند.
- **پلن‌های قیمت‌گذاری، متن Hero/CTA و لوگو** همگی به‌صورت زنده از پنل
  مدیریت پلتفرم خوانده می‌شوند (با مقادیر پیش‌فرض ثابت به‌عنوان
  fallback) — به بخش [مدیریت پلتفرم](#مدیریت-پلتفرم) مراجعه کنید.
- یک **کلید واقعی تعویض قیمت ماهانه/سالانه** — با تعویض، اعداد واقعی هر
  پلن تغییر می‌کند (نه فقط تعویض یک برچسب ظاهری)، به‌همراه یادداشت
  «صورتحساب سالانه».
- **کاملاً بهینه‌شده برای موبایل**: یک منوی همبرگری واقعی (نه یک نوار
  پنهان) تمام لینک‌های ناوبری، تعویض‌کننده‌ی زبان و دکمه‌های ورود/دمو/
  ثبت‌نام را زیر breakpoint سایز `lg` نمایش می‌دهد، به‌همراه یک **نوار تب
  پایینی** برای دسترسی سریع با انگشت شست به بخش‌های امکانات، ماژول‌ها،
  قیمت‌گذاری و دکمه‌ی «امتحان دمو» هنگام اسکرول (دکمه‌ی «منو»ی آن همان
  پنل کامل منوی همبرگری را باز می‌کند)؛ همین‌طور برای صفحه‌ی ورود که پنل
  برند آن در موبایل روی فرم قرار می‌گیرد نه اینکه ناپدید شود.
- **فضای کاری داخل برنامه** هم برای موبایل بهینه شده: Module Rail — که در
  دسکتاپ همیشه نمایش داده می‌شود و بین حالت فقط-آیکون و حالت کامل با
  برچسب متن قابل باز/بسته شدن است (ترجیح کاربر بین نشست‌ها ذخیره
  می‌شود) — در موبایل به یک کشوی کشویی تبدیل می‌شود (با یک آیکون همبرگری
  در Topbar باز می‌شود) که همان قابلیت باز/بسته شدن را دارد، و هر جدول
  داده به‌جای بریده‌شدن به‌صورت افقی اسکرول می‌شود.
- **نوار زیرمنو هنگام جمع‌بودن ساید‌بار** — جمع‌کردن Module Rail به حالت
  فقط-آیکون، زیرصفحه‌های آکاردئونی را که معمولاً فهرست می‌کند پنهان می‌کند؛
  بدون جایگزین، کاربری که نداند باید دوباره بازش کند عملاً دسترسی راحت به
  اکثر گزینه‌های آن فضای کاری را از دست می‌دهد. یک نوار ناوبری افقی این
  خلأ را خودکار پر می‌کند: تا وقتی ساید‌بار جمع است، فهرست کامل زیرصفحه‌های
  فضای کاری فعال (مخاطبین، شرکت‌ها، سرنخ‌ها، …) به‌صورت یک ردیف بالای
  محتوا ظاهر می‌شود؛ به‌محض باز شدن ساید‌بار دوباره ناپدید می‌شود، تا همان
  لینک‌ها هرگز دوبار نمایش داده نشوند.
- **ناوبری آکاردئونی نوار کناری** — زیرصفحات هر فضای کاری (مخاطبین،
  شرکت‌ها، سرنخ‌ها، معاملات، …) مستقیماً به‌عنوان یک بخش قابل‌بازشدن داخل
  Rail قرار می‌گیرند، نه یک نوار جدا در بالا؛ بخش فعال به‌طور خودکار باز
  می‌شود. هشت ماژول صنعتی کسب‌وکار زیر یک گروه **قالب‌ها** قابل‌جمع‌شدن
  گردآوری شده‌اند که جهت‌گیری قالب‌های صنعتی را منعکس می‌کند (به
  [قالب‌های صنعتی](#استودیوی-کم‌کد) مراجعه کنید)، به‌جای شلوغ‌کردن ناوبری
  اصلی.
- **راهنمای هر قابلیت** — یک دکمه‌ی «؟» کنار عنوان تقریباً هر صفحه، یک
  توضیح کوتاه (چگونگی کارکرد قابلیت، چرا مفید است) باز می‌کند که به زبان
  فعال فضای کاری بومی‌سازی شده؛ به `helpTopics` در پیام‌های i18n و
  `src/components/patterns/feature-help.tsx` مراجعه کنید.
- هر دیالوگ در صفحه‌های باریک یک حاشیه‌ی کناری راحت (`calc(100%-2rem)`
  به‌جای `100%` لبه‌به‌لبه) حفظ می‌کند — یک اصلاح کوچک و سیستمی که به نفع
  هر modal در اپلیکیشن روی موبایل است.

### برند و بین‌المللی‌سازی

- نام و نشان محصول: **SOBI CRM**، با یک لوگو/فاویکون واقعی که به‌طور
  یکدست در نوار اصلی فضای کاری، پنل برند صفحه‌ی ورود/ثبت‌نام، صفحه‌ی
  عمومی قرارداد و لندینگ‌پیج استفاده می‌شود.
- سه زبان — انگلیسی، آلمانی، فارسی — با پشتیبانی کامل از راست‌به‌چپ برای
  فارسی.
- **Vazirmatn** فونت زبان فارسی در همه‌جا است: در فضای کاری اصلی (از
  طریق تعویض متغیر CSS `:lang(fa)` برای `--font-ui`) و لندینگ‌پیج (از
  طریق یک تعویض معادل `--landing-font-*`).
- پشتیبانی از تقویم شمسی برای شماره‌گذاری قرارداد و گزارش درآمد.
- **هر بخش از فضای کاری یک دکمه‌ی راهنمای وابسته به زمینه دارد** — یک
  «؟» کنار عنوان صفحه (از طریق الگوی مشترک `PageHeader`/`FeatureHelp`)
  که با کلیک، دیالوگ کوتاهی باز می‌کند و توضیح می‌دهد این ویژگی چه‌کاری
  انجام می‌دهد و چرا مفید است، به زبان فعال. این محدود به صفحات اصلی CRM
  نیست: مدیریت (تنظیمات، حسابرسی، سلامت، یکپارچه‌سازی‌ها، ماژول‌ها، نقش‌ها،
  کاربران)، فضای کاری و دستیار هوش مصنوعی، هر سازنده‌ی استودیو (موجودیت‌ها،
  اتوماسیون‌ها، گردش‌کارها، Rules، قالب‌ها)، پشتیبانی، صورتحساب، پروفایل و
  هر هشت ماژول صنعتی را نیز پوشش می‌دهد — یک بازبینی کامل، دقیقاً همان
  شکاف‌هایی را که در بازبینی دستی کیفیت پیدا شد بست (سازنده‌ی گردش‌کار و
  گراف روابط پیش از این بازبینی، دو مورد آشکارترین کمبود بودند).
- **بدون رشته‌ی زبانی هارد‌کدشده** — هر برچسب، هر Chip وضعیت، هر مقدار
  enum و حتی محتوای دمو/جای‌گیر از طریق next-intl برای زبان فعال حل
  می‌شود، نه اینکه در یک زبان ثابت شده باشد. این مورد در یک بازبینی
  سراسری برنامه، در دو نقطه به‌عنوان یک باگ واقعی پیدا و رفع شد: صفحه‌ی
  دستیار هوش مصنوعی صرف‌نظر از زبان انتخاب‌شده متن ثابت فارسی نمایش
  می‌داد، و تیکت دموی مرکز پشتیبانی نیز همین‌طور بود — هر دو اکنون در هر
  سه زبان به‌درستی ترجمه می‌شوند.
- هرجا یک تنظیم یا برچسب منو معادل فارسی یا آلمانی رایج و طبیعی داشته باشد
  (به‌جای ترجمه‌ی تحت‌اللفظی و ناهموار)، ترجمه‌های فارسی و آلمانی همان
  معادل رایج را به کار می‌برند.

### حالت دمو

- یک دکمه‌ی **یک‌کلیکی «ورود به فضای کاری دمو»** در صفحه‌ی ورود و
  لندینگ‌پیج که با حساب دموی عمومی و فقط‌خواندنی
  (`public-demo@sobi.local`، فضای کاری «Sobi CRM Demo») وارد می‌شود —
  بدون نیاز به ثبت‌نام برای امتحان محصول.
- **در build پروداکشن غیرفعال است** — تأیید شده که کاملاً حتی رشته‌ی
  اعتبارنامه از باندل کلاینت پروداکشن حذف می‌شود.
- اسکریپت seed یک مستأجر دموی واقع‌گرایانه را با شرکت‌ها، مخاطبین،
  معاملات، سرنخ‌های باز و تبدیل‌شده، یک مقاله‌ی پایگاه دانش، یک قرارداد
  ارسال‌شده و یک کمپین پیش‌نویس پر می‌کند.

## استقرار

برای متغیرهای محیطی، گزینه‌های پلتفرم و چک‌لیست انتشار به
[`docs/DEPLOYMENT.md`](docs/DEPLOYMENT.md) مراجعه کنید. نکات کلیدی برای
پلتفرم‌های سرورلس (Vercel و مشابه):

- `DATABASE_URL` باید به یک **Postgres مدیریت‌شده و در‌دسترس** اشاره کند
  (Neon، Supabase، RDS، …) — Postgres داکر محلی توسعه از یک محیط
  دیپلوی‌شده در‌دسترس نیست. اگر ارائه‌دهنده‌ی شما اتصال‌ها را از طریق یک
  pooler عبور می‌دهد (مثلاً pgbouncer در Supabase)، `DIRECT_URL` را هم به
  یک اتصال **بدون pooler** تنظیم کنید — `prisma migrate deploy` به آن نیاز
  دارد، چون حالت تراکنشی pooler از مایگریشن اسکیما پشتیبانی نمی‌کند.
- `BETTER_AUTH_SECRET`، `BETTER_AUTH_URL`، `NEXT_PUBLIC_APP_URL` و
  `FIELD_ENCRYPTION_KEY` را روی مقادیر واقعی پروداکشن تنظیم کنید.
- **درایور ذخیره‌ی فایل روی دیسک محلی فقط روی یک میزبان دائمی تک‌نمونه
  کار می‌کند** — در پلتفرم‌های سرورلس، پیش از تکیه بر آپلود فایل،
  درایور داخلی را با `FILE_STORAGE_DRIVER=s3` و تنظیمات bucket فعال کنید.
- در production، `FILE_STORAGE_DRIVER=s3` و `RATE_LIMIT_BACKEND=redis` الزامی
  هستند؛ eventها از outbox پایدار PostgreSQL و jobهای دارای lease/retry استفاده می‌کنند.
- `npx prisma migrate deploy` باید پیش از (یا به‌عنوان بخشی از) اولین
  دیپلوی روی دیتابیس پروداکشن اجرا شود.
